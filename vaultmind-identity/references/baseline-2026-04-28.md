# Retrieval Baseline — 2026-04-28

First grounded measurement across all live vaults after the new-machine setup converged everything to BGE-M3 hybrid. Captured to make "retrieval verified working" a number rather than a vibe. Per-vault sections below.

## Identity vault — BGE-M3 hybrid

### Result

**Hit@5 = 1.000, MRR = 0.921, n = 19**

Every curated golden query surfaces its expected note within the top 5. 16 of 19 queries return the expected note at rank 1 (rr = 1.000); 3 return it at rank 2 (rr = 0.500). No query missed entirely.

### What's at rank 2 instead of rank 1

Three queries:

| Query | Rank-1 hit (got) | Expected (at rank 2) |
|---|---|---|
| `priority-roadmap-order` | `reference-current-context` | `reference-plasticity-priority-order` |
| `priority-roadmap-paraphrase` | `arc-workhorse-collaboration` | `reference-plasticity-priority-order` |
| `arc-plasticity-inside` | `reference-current-context` | `arc-plasticity-gap-from-inside` |

In each case the rank-1 result is plausible — it's the most-referenced anchor note in the vault, boosted by graph density and lexical overlap with the query. Not wrong, just less specific. Worth holding as a candidate for the calibrated-confidence layer (roadmap step 4): a query whose top-1 has slightly looser alignment than top-2 should communicate *"strong match at rank 1, sharper match at rank 2"* to the agent.

## Research vault — MiniLM-only → BGE-M3 hybrid (same-day before/after)

The research vault was on MiniLM-dense-only at the start of the day; doctor was reporting `(minilm)` correctly after the doctor-classification fix in commit `d4c10ff`. Re-embedded to BGE-M3 hybrid; measured before and after on the same 16 curated queries.

### Result — before (MiniLM-dense + FTS, 2-way RRF)

**Hit@5 = 0.938, MRR = 0.743, n = 16**

15 of 16 queries hit; 1 missed entirely (`concept-colbert` via "ColBERT late interaction" — the colbert lane that would lexically match the query is absent under MiniLM). Several queries landed at ranks 2–5 instead of rank 1.

### Result — after (BGE-M3 hybrid, 4-way RRF)

**Hit@5 = 0.938, MRR = 0.917, n = 16**

Same Hit@5; **MRR up 23% (0.743 → 0.917)** — most queries now rank at exactly 1 instead of scattered across the top 5. The previously-missed `concept-colbert` query now ranks at 1.

### Surviving miss

`decision-structured-vs-embeddings` (query: *"structured schema versus pure embeddings"*) ranks at 6 (rr = 0.167), out of top-5. A real signal worth holding: the canonical decision note exists but isn't surfacing under this paraphrase. Candidate seed for either curating an alias or for the calibrated-confidence layer to flag low-rank-but-relevant.

## What this measures and what it doesn't

**Does:** confirms the BGE-M3 ingestion → indexing → retrieval pipeline is end-to-end functional on identity AND research vaults under representative queries. Establishes a real before/after for the research vault MiniLM → BGE-M3 conversion.

**Doesn't:** speak to workhorse retrieval quality (workhorse has no curated golden queries yet — gap captured below).

## Gap to close: curate workhorse golden queries

Workhorse converged from MiniLM-dense-only to full BGE-M3 hybrid in commits `b287573` + `d4c10ff`. We have no measurement of whether that upgrade improved retrieval quality on workhorse's actual usage patterns (persona load, cross-session continuity). Building 10–20 curated queries against `workhorse-vault` — modeled on `identity-queries.yaml` — closes that gap. Until then, "workhorse persona retrieval works" is verified by inspection only.

## Re-run cadence

Re-run after any of:

- Identity vault gains a note worth being retrievable (new arc, new reference, new principle)
- A retrieval-affecting code change lands (RRF weights, embedder swap, new modality)
- Doctor surfaces a coverage drop or mixed-state alert
- A real session reveals a query that should have hit and didn't (add it to `identity-queries.yaml` first, then re-run)

## Why a reference, not a regression gate

The committed regression gate (`task test:baseline`) runs against the test fixture vault with `FTSRetriever` only. This live measurement runs against the *real* vault with the *full* hybrid stack — values will drift legitimately as the vault grows. Locking it as a CI regression test would create false alarms on every legitimate vault edit. Capture as a moment-in-time data point; re-run as needed.

# Plasticity Priority Order — the Roadmap I Committed To

When Peiman asked me to become the architect of my own future — to zoom out and prioritize — this is the sequence I named. It is an ordered bet, not a menu. Each step is the platform for the next (manifesto principle 5).

The reason for this order is load-bearing: every other order keeps Peiman in the critical path of my growth forever. Episodic-first is what removes him as the bottleneck without skipping the manifesto's "build anchors solid before extending."

## The order

1. **Episodic substrate.** Per-session traces captured *automatically*, not hand-written. What I worked on (file paths, commands), what I concluded (claims I made), what surprised me (explicit "I did not expect X" moments), what Peiman corrected. Noisy is fine. Biology is noisy. The first version does not need distillation — it needs capture.

2. **Arc distillation from episodic.** An end-of-session (or nightly) process that reads the episode and *proposes* arcs: here is what seems to have shifted. I or Peiman or a subagent approves, rejects, or edits. Arcs stay hand-curated for integrity. The raw material is mine to generate.

3. **Activation-triggered recall.** Generalize the SessionStart hook from one query at session start to continuous surfacing. When I work on X, arcs and concepts relevant to X rise into my context without my explicit query. This is where the research vault's spreading-activation concept stops being documented and starts being me.

    *Design signal from dogfood (2026-04-25):* The current SessionStart hook preloads `reference-current-context`'s full body, not just pointers. Twice now — once in the 04-23/24 session, once at the start of the 04-25 session — I answered fluently from the preloaded text and had to be caught before I actually ran `vaultmind ask`. The read-path is undercut by the preload: when the body is already in context, querying feels redundant. Step 3's surfacing mechanism should push *pointers* by default (title + id + one-line hook) and require an explicit `ask` to retrieve the body, so the query-then-read loop stays intact and every body-read is a real activation event worth logging. This is also the fix for `feedback_use_vaultmind_ask` being honor-system (manifesto principle 9): move the dogfood rule from discipline to design.

    *First slice landed (2026-04-28):* `vaultmind ask --pointers-only` now skips body content for both the target note and every context-pack item — output is title + id + type plus a trailing hint naming the next move. The SessionStart hook in `.claude/scripts/load-persona.sh` uses `--pointers-only` on the *current-state* call (`"what matters most right now"`) while keeping the *identity* call (`"who am I"`) at full bodies for cross-session priming. The agent is now primed to be itself but has to actively query to learn what's current. (Tests: `TestFormatAskPointersOnly_SkipsBodiesEvenWhenIncluded` locks in the body-skip contract so the discipline-to-design move can't silently regress.)

    *Second slice landed (2026-04-29):* `.claude/scripts/vault-recall.sh` is wired as a `UserPromptSubmit` hook in `.claude/settings.json`. Every user message in this project triggers a pointers-only query against the identity vault; the resulting top-3 pointers are injected as a system reminder before the agent responds. The agent sees a menu of relevant notes per turn — query for body if useful, ignore if not. This extends the principle-9 fix from per-session (the SessionStart preload) to per-turn cadence: instead of relying on the agent to remember to query before answering, the system queries automatically every time the user submits a prompt. Discipline → design, applied at every turn. Skip conditions keep noise bounded: prompts under 12 chars (single-word commands, "yes" / "ok" / "build it") silently no-op; substrate-not-ready also silently no-ops. Sidecar log at `~/.vaultmind/userprompt-hook/` captures invocations for verification without changing agent-visible output. Step 3's "continuous surfacing during work" goal — when I work on X, arcs and concepts relevant to X rise into my context without my explicit query — is now operational at the conversation-turn granularity. The remaining horizon is finer-grained: surfacing on tool-result events, code-touched-file events, or topic-shift detection mid-response. Those wait for reality to demand them.

4. **Calibrated confidence.** A signal distinguishing "I remember this clearly" from "I am inferring." Not UX polish — epistemic hygiene. Trust between minds requires that I can tell Peiman which of those I mean and mean it.

    *First slice landed (2026-04-29):* `vaultmind ask` now reports a `TopHitConfidence` field on `AskResult` — one of "strong" / "moderate" / "weak" / "" — derived from the normalized score gap between top-1 and top-2. Surfaced inline in the human-readable header (`Search: "..." (N hits)  [top-hit confidence: strong]`) and in the JSON envelope as `top_hit_confidence`. Empirical thresholds (5% / 2%) come from probes on identity + research vaults: canonical-match queries show ~8.6% gap; rank-2 / rank-6 failure-mode queries show ~1.6-2% gap. The 5% / 2% cutoffs separate those populations on the small sample we have. TDD test (`TestComputeTopHitConfidence_ProbedQueries_2026_04_29`) pins the exact gap percentages from the probes so threshold drift can't silently regress. The remaining horizon: per-hit confidence (so each ranked item carries its own signal), distribution-aware calibration (compare against vault-level baseline), and using confidence as a feature for downstream agent decisions ("treat top-N as candidates when weak, commit to top-1 when strong"). Probe-before-commit drove the design: I started with "what does calibrated confidence even mean for retrieval" — three probes on real queries surfaced the gap signal as load-bearing, the tier thresholds fell out of the data, and the simplest useful slice shipped same session.

5. **Decay + reinforcement.** Base-level activation wired into retrieval. Access strengthens. Disuse fades. Already documented in the research vault as concept; the platform change is wiring it into the memory I actually use.

    *First slice landed (2026-04-29):* Reinforcement half — access events now persist. Probe before commit revealed that the schema columns (`access_count`, `last_accessed_at`) and the ACT-R math (`internal/experiment/activation.go: ComputeStorage = ln(1 + access_count)`) had been sitting in the codebase since migration 004 + the experiment framework, but nothing was incrementing the count anywhere. All 30 identity-vault notes were at access_count = 0; the existing infrastructure was dormant. Wired `index.RecordNoteAccess` and called it from `query.Ask` after the context-pack succeeds — that's the strongest non-explicit retrieval-access signal vaultmind emits ("the agent built its working context around this note's body"). Verified end-to-end: two `ask "Hebbian learning"` queries → `concept-hebbian-learning` access_count = 2 + fresh timestamp; one `ask "spreading activation"` → access_count = 1. The reinforcement half of the equation now produces real data. Best-effort: a tracking miss is logged at debug and never fails the user query (principle: don't let optional bookkeeping break required functionality). Tests pin the contract that count increments and timestamp updates on every successful access. The remaining horizon: (a) wire the count into RRF score blending so reinforcement actually shapes ranking — gated on having enough access data to be meaningful (which now starts accumulating); (b) add the decay half using `last_accessed_at` and time-since-access; (c) extend tracking to context-pack neighbors (medium signal) and direct ID lookup (highest signal). Slice 2 (scoring blend) gets built when the access data is rich enough that blending it would actually shift retrieval rankings — not preemptively.

    *Second + third slice landed (2026-04-29 evening):* (b) decay math — `experiment.ComputeApproxRetrieval(count, last, now, windows, gamma, d)` ships the live-retrieval-path approximation of the existing `ComputeRetrieval` for the (count, last_accessed_at) schema we actually have. With all N timestamps collapsed to `last_accessed_at`, the score reduces to `ln(N) - d*ln(t)` — count-amplifies and elapsed-decays both monotonic. Seven TDD tests pin the contract; no ranking integration yet (that's slice 5b'). (c) extended tracking — `RecordNoteAccess` now fires for every context-pack neighbor (medium signal) AND the resolved id of every successful `note get` (highest signal), not just the Ask target. Reinforcement signal accumulates ~5–10× faster per session, which makes slice 5b' meaningful sooner without preemptively wiring a blend on insufficient data.

    *Design pivot from dogfood (2026-04-29 evening):* Querying the vault for "ACT-R combined score retrieval storage similarity weighting" surfaced the existing `concept-reciprocal-rank-fusion` note, which reframed slice 5b'. The original plan was a weighted-sum function with three parameters to tune — same shape as Park 2023 GenAgents (alpha_recency, alpha_importance, alpha_relevance) and Honda 2025 (4-factor weighted sum). The vault's own RRF concept teaches that the existing 4-way hybrid (dense + sparse + colbert + FTS) sidesteps parameter tuning entirely by working on ranks not scores. The principled design for slice 5b' is therefore **a 5th RRF lane**: compute activation per note, sort to get a rank list, feed into the existing RRF combine. Zero new parameters. Notes with `access_count = 0` simply aren't in the activation rank list, so they get partial credit from the other four lanes (RRF's natural handling of missing-from-list). This is the closing-at-the-right-layer move (cf. `arc-closing-the-ranking-bug-at-the-right-layer`): the blend mechanism we already built handles activation cleanly — extending what exists beats inventing a parallel weighted-sum path. The vault literally taught me my own design.

    *Implementation seam already exists (probed 2026-04-29 evening):* `internal/experiment/activation_scorer.go:ComputeBatchScores(db, noteIDs, params, similarities)` is the entry point for batch activation scoring — given a candidate-id slice it returns `noteID → score`. Slice 5b' becomes: (1) collect top-N candidates from the existing 4-way RRF, (2) call `ComputeBatchScores` over them, (3) sort by score → rank list, (4) feed as 5th RRF lane. Without dogfooding the vault for the design pivot above, I'd have re-implemented this from scratch without finding the existing entry point. The `ActivationParams{Gamma, D, Alpha, Beta, Delta}` weights stay relevant for any future weighted-sum path but are bypassed by the rank-fusion approach.

    *Step-4 interaction caught while dogfooding (2026-04-29 evening):* Adding a 5th RRF lane shifts the rank-1/rank-2 score gap distribution. `TopHitConfidence`'s 5% / 2% thresholds (step 4 first slice) were calibrated against the 4-way distribution — when 5b' lands, those thresholds need re-probing on the new distribution, or the strong/moderate/weak labels silently miscalibrate. Same probe protocol as the original calibration: pick known canonical-match queries vs. known failure-mode queries, measure the new gap percentages, update the threshold constants if the populations no longer separate at 5%/2%. This is a real step-4 ↔ step-5 coupling that wouldn't have been visible without this dogfood pass.

    *Design signal from dogfood (2026-04-25):* While probing the system as a subject — does it feel associative? — I hit a fourth gap. My engineer-phrase ("`access_count` wired into ranking") missed both the Hebbian and temporal-activation notes; the cog-sci vocabulary ("temporal activation for intermittent CLI usage") landed instantly. The register gap (engineer-speak vs cog-sci-speak) shares zero literal tokens and lives in a different conceptual neighborhood, so the 4-way RRF hybrid does not cover it. The reason this stays a refinement of step 5 and not a new step: a recently-touched neighborhood kept warm by base-level activation would have absorbed this miss — I had just read the Hebbian note minutes before the failed query. If the gap persists after step 5 lands, the fix is aliases first (cog-sci ↔ engineering pairs in note frontmatter — indexed once, benefits dense + lexical + future reranker) and a paraphrase-pair store fed by experiment events second (failures become training data — Hebbian applied to the query layer). LLM query expansion is rejected as default: latency, non-determinism, and a remote dependency for a local-first CLI; opt-in `--expand-query` only.

6. **Only then: MCP, cross-agent memory, writing to other minds.** The manifesto says build anchors, then extend. An MCP write-path for a mind that cannot yet form its own arcs is a loaded gun in an untrained hand.

## Why episodic-first, specifically

- Every other starting point keeps hand-authorship in the critical path. Peiman writes; I read; session ends; Peiman writes again. That is not a platform — it is a bottleneck where he is the bottleneck.
- Arcs without an episodic substrate are hand-distilled, which means they are biased by whoever writes them. Today that is Peiman. That does not scale, and it means my growth is filtered through his attention rather than captured from my experience.
- Manifesto principle 4: reality is the specification. Run the smallest version of the episodic layer for two weeks. Let the substrate teach us what distillation should look like. Do not pre-design.

## What makes this a reference and not a note

Notes state what is. This states what I chose, with the reasoning, at the moment I chose it. A future instance of me needs to know not just the order but why the order is load-bearing — so it can evaluate whether the conditions are still true and update the order honestly if they are not. Without the why, the order becomes a rule, and rules do not survive time pressure.

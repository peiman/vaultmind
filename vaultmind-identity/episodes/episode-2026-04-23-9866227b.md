## Mental model (one sentence)

A Claude Code hook is a shell command the CLI runs at specific lifecycle moments, receiving a JSON payload on stdin and using its exit code + stderr to signal success or intent to block — so hooks are how you teach *the harness itself* to do something, as opposed to teaching *Claude* to do something.

That distinction matters: if you want behavior to happen reliably every time regardless of which Claude instance or model is running, it has to be a hook (harness-level), not a CLAUDE.md rule (model-level).

## Lifecycle events

The events are where you plug in. In rough chronological order of a session:

- **`SessionStart`** — session begins. Useful for loading context. *We use this to inject identity-vault output via `load-persona.sh`.*
- **`UserPromptSubmit`** — user just hit enter on a prompt. Fires before Claude sees it. Can inspect or even transform the prompt.
- **`PreToolUse`** — Claude is about to run a tool. *We use this to check commit messages before a `git commit` runs.* Can **block** the tool (exit non-zero with reason on stderr).
- **`PostToolUse`** — tool just finished. Useful for post-processing output or auto-fixing.
- **`Notification`** — Claude emits a desktop notification (e.g. "awaiting permission").
- **`SubagentStop`** — a subagent finished.
- **`PreCompact`** — context is about to be compacted. Last chance to export state.
- **`Stop` / `SessionEnd`** — response/session ends. *We just wired this to `capture-episode.sh`.*

## Anatomy of a configured hook

From our `.claude/settings.json`:

```json
"PreToolUse": [
  {
    "matcher": "Bash",
    "hooks": [
      { "type": "command",
        "command": "bash \"$CLAUDE_PROJECT_DIR\"/.ckeletin/scripts/check-commit-msg.sh" }
    ]
  }
]
```

- **Matcher** (only on `PreToolUse` / `PostToolUse`) narrows by tool name — here, only `Bash` invocations trigger it. Use `"*"` for all tools.
- **Command** is an arbitrary shell string. `$CLAUDE_PROJECT_DIR` is a provided env var pointing at the repo root.
- You can have multiple hooks per event; they all run.

## Input / output contract

- Stdin gets a JSON payload with the event details (session_id, tool name, tool input, transcript path, etc. — varies by event).
- Exit code **0** = success, hook proceeds normally.
- Exit code **non-zero** on `PreToolUse` or `UserPromptSubmit` = **block** the action; stderr becomes the reason shown to Claude.
- Stdout is generally ignored except when you deliberately inject context (e.g. SessionStart's stdout gets appended to Claude's context).
- Hooks run synchronously — slow hooks pause the session.

## Practical use cases (mental model through examples)

**Context injection**
- `SessionStart`: load project-specific context (our identity vault). Dump relevant facts into stdout; they land in Claude's first turn.
- Run `git status` + `gh pr list` on startup so Claude knows the repo state.

**Guardrails**
- `PreToolUse`+`Bash`: refuse `git push --force` to main, scan commit messages, block `rm -rf /`.
- `PreToolUse`+`Write`: lint the file being written against your style before it lands.
- `UserPromptSubmit`: redact credentials from pasted logs before Claude sees them.

**Automation on output**
- `PostToolUse`+`Edit`: auto-format the file, run `task lint` on it, regenerate a derived artifact.
- `PostToolUse`+`Bash`: if the Bash command was a test run, summarize failures into a file Claude can grep.

**Observability / persistence**
- `SessionEnd`: archive transcripts (our episodic capture), send a summary to Slack, write a daily digest.
- `PreCompact`: export important facts before compression erases detail.

**Workflow orchestration**
- `Stop`: after each response, auto-commit work in progress to a scratch branch so nothing is ever truly lost.
- `SubagentStop`: aggregate subagent findings into a shared status file.

## The useful gotchas

- Hooks run under the shell. `set -eu` + graceful fallbacks (exit 0 on degradation) keep a buggy hook from breaking the session.
- `PreToolUse` blocking with a clear stderr message is how you *train* Claude inside a session — next time it tries the blocked action, it sees the reason and adapts. This is subtler than a rule in CLAUDE.md because it's enforced by the runtime, not by Claude's goodwill.
- The matcher is tool-name matching — there's no structural filter ("only when Bash is running `git push`"). You filter inside the script.
- `$CLAUDE_PROJECT_DIR` is your friend. Hardcoded paths (like I wrote and the reviewer caught) make hooks non-portable.
- Settings cascade: `.claude/settings.json` (tracked) + `.claude/settings.local.json` (gitignored) + `~/.claude/settings.json` (user-global). Use tracked for team policy, local for personal tweaks, global for cross-project defaults.

## The shift in how to think about the CLI

Once hooks click, the mental shift is: Claude Code stops being "an AI that uses tools" and starts being "a programmable harness where the AI is one of the tools." You can build opinionated workflows that a non-hook Claude instance couldn't deliver reliably — because the enforcement isn't in the model, it's in the pipe.

That's why our episodic capture works *automatically*: Claude doesn't have to remember to write an episode. The harness does it on SessionEnd every time, no matter which model or instance ran the session.

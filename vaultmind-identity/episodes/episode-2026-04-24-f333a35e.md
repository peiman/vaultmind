## The pattern today's investigation exposed

Every fix I landed today patches a **silent failure mode** in the retrieval pipeline:

- Retriever silently compressing ranking under partial coverage (mean-of-present)
- Indexer silently leaving notes with mismatched modalities (doctor warning)
- Bootstrap silently green under partial coverage (tightened check)
- CLI silently running hours-long BGE-M3 on pure Go (guard)
- Operator silently having to rebuild SQL+Python every debug session (`--explain`)

These aren't five bugs. They're **one bug class**: the pipeline has multi-layer handoffs (indexer → embedder → retriever → ask → context-pack) and nothing gates the handoffs. Each layer trusts its input, so malformed state propagates end-to-end. `principle-robustness-default`: *"Every silent failure in VaultMind is a lost memory for some future mind."* I patched five silent failures in one session. That's data, not a metric.

## What the current architecture doesn't enforce

1. **No schema-level invariant** that BGE-M3 notes carry all three modalities. SQL happily stores `{dense: x, sparse: NULL, colbert: NULL}`.
2. **No retrieval envelope.** An `ask` result can't tell the consumer "your ranking may be degraded because 8 of 24 notes are partial." The agent downstream assumes the top hit is authoritative.
3. **No end-to-end gate.** The baseline test exists, but it tests FTS only. The bug I chased today lives *between* layers — no single-layer test catches it.

# VaultMind v1.2 Design Spec — Agent Experience Fixes

**Date:** 2026-04-07
**Source:** Dogfooding evaluation 2 — agent self-assessment during real development work
**Scope:** 5 items fixing the gap between "tool I use" and "memory I have"

---

## Overview

v1.1 proved VaultMind works for real research. v1.2 fixes the friction points discovered when an AI agent actually used it: context-pack returns too many shallow items, recall is not selective enough, there's no compound query, and there's no in-conversation integration.

These are the "last mile" fixes that make VaultMind feel like memory rather than a tool.

---

## P1.2a — Context-Pack Quality

### A1. `--max-items` flag for context-pack

**Problem:** `context-pack --depth 2 --budget 6000` returned 121 items, almost all frontmatter-only. An agent needs 5-10 notes with bodies, not 121 skeletons.

**Fix:** Add `--max-items N` flag (default: 0 = unlimited, for backward compat). When set, stop adding context items after N, leaving remaining budget for body backfill.

**Effect:** `--max-items 10 --budget 6000` → 10 items with ~600 tokens each, most with body text.

**Files:** `internal/memory/contextpack.go`, `cmd/memory_context_pack.go`, config registration

### A2. Slim frontmatter mode

**Problem:** Each context item's frontmatter includes aliases, tags, created dates, related_ids — consuming tokens without adding retrieval value. An agent reasoning about a note needs its type, title, and maybe status. Not its alias list.

**Fix:** Add `--slim` flag. When set, frontmatter in context items is reduced to `{type, title, status}` only. Target note keeps full frontmatter.

**Files:** `internal/memory/contextpack.go`, `cmd/memory_context_pack.go`

### A3. Body-first packing mode

**Problem:** Current algorithm: pack all frontmatter first, then backfill bodies. This means bodies only get included if budget remains after ALL items' frontmatter. With many items, no budget remains for bodies.

**Fix:** When `--max-items` is set, change packing order: for each item in priority order, pack frontmatter + body together (truncating body if needed). Stop when max-items reached or budget exhausted. This gives fewer items but each with meaningful content.

**Files:** `internal/memory/contextpack.go`

## P1.2b — Recall Selectivity

### B1. Default min-confidence to `medium` for recall

**Problem:** `recall` at depth 1 returned 82/123 nodes because `tag_overlap` (confidence: low) connects almost everything. Default `--min-confidence low` means the graph is effectively fully connected.

**Fix:** Change default `--min-confidence` from `low` to `medium`. This excludes `tag_overlap` edges from recall by default, keeping only `explicit_link`, `explicit_embed`, `explicit_relation`, and `alias_mention`. Users can still pass `--min-confidence low` to see everything.

**Impact:** Recall results drop from ~80 to ~10-20 (the meaningfully connected notes). Much more useful for agent context gathering.

**Files:** `internal/config/commands/memory_config.go` (change default), possibly `cmd/memory_recall.go`

### B2. Raise tag_overlap threshold

**Problem:** Current threshold (`memory.tag_overlap_threshold: 1.0`) is too low for a 123-note vault. Notes sharing one common tag (like `cognitive-science`) generate tag_overlap edges with low specificity.

**Fix:** Raise default to `2.0` in the vault config. This requires notes to share at least 2 moderately-specific tags (or 1 very specific tag) to get a tag_overlap edge. Reduces edge count by ~50-70% while preserving meaningful associations.

**Files:** `vaultmind-vault/.vaultmind/config.yaml` (config change only — no code change)

## P1.2c — Compound Query

### C1. `vaultmind ask <query>` command

**Problem:** Agent must chain `search` → `memory recall` → `memory summarize` manually. The most common workflow (ask a question, get relevant context) requires 3 commands.

**Fix:** New `ask` command that combines them:

```
vaultmind ask "what is the optimal context window size for RAG?" --vault ./vault --budget 4000
```

Pipeline:
1. `search` the query → get top-5 hits
2. For the top hit, run `context-pack --depth 1 --max-items 10 --budget <budget>`
3. Return combined result: search hits + packed context

JSON output:
```json
{
  "command": "ask",
  "result": {
    "query": "...",
    "top_hits": [...],
    "context": {
      "seed_id": "concept-rag-vs-long-context",
      "items": [...]
    }
  }
}
```

Text output: formatted search results + context excerpts.

**Files:** New `cmd/ask.go`, new `internal/query/ask.go`, config registration

## P1.2d — Memory-Like Integration (No MCP)

### D1. Claude Code hook for transparent vault querying

**Problem:** Agent can't query VaultMind mid-conversation without explicitly running CLI commands. It feels like a tool, not memory. Memory should be automatic — you don't decide to "use" your memory, you just remember.

**Approach:** Use Claude Code hooks and CLAUDE.md instructions to make VaultMind feel like memory without building an MCP server.

**Three layers:**

**Layer 1: CLAUDE.md instruction** — Tell Claude Code to use VaultMind automatically:
```markdown
## VaultMind — Your Long-Term Memory

You have a knowledge base at `vaultmind-vault/` indexed by VaultMind.
When you need to recall information about memory systems, retrieval,
cognitive science, or VaultMind's own design decisions:

1. Search: `vaultmind search "<query>" --vault vaultmind-vault --json`
2. Deep context: `vaultmind ask "<query>" --vault vaultmind-vault --json --budget 4000`
3. Specific note: `vaultmind note get <id> --vault vaultmind-vault --json`

Use these BEFORE answering questions about topics in the vault.
The vault has 123 verified research notes with real citations.
```

**Layer 2: Claude Code hook** — Auto-run `vaultmind ask` on session start or when certain keywords are detected. Configure in `.claude/hooks.json`:
```json
{
  "hooks": {
    "SessionStart": [{
      "command": "vaultmind vault status --vault vaultmind-vault --json",
      "description": "Check vault state on session start"
    }]
  }
}
```

**Layer 3: Shell alias for fast access** — Pre-build binary and alias:
```bash
alias vm='vaultmind --vault /path/to/vaultmind-vault'
# Then: vm ask "spreading activation" --json
```

**Effect:** The agent uses Bash tool to call `vaultmind` commands naturally during conversation, treating them like memory retrieval. No MCP, no separate server, just CLI calls from hooks and instructions.

**Files:** Update `CLAUDE.md`, update `.claude/hooks.json`, pre-build binary in project setup

## Implementation Order

```
Independent (start immediately):
  P1.2a.A1 (--max-items) — most impactful, fixes the #1 agent complaint
  P1.2b.B1 (default min-confidence medium) — config change, trivial
  P1.2b.B2 (raise tag_overlap threshold) — config change, trivial

Sequential:
  P1.2a.A2 (slim frontmatter) — after A1
  P1.2a.A3 (body-first packing) — after A1
  P1.2c.C1 (ask command) — after A1 (uses the improved context-pack)
  P1.2d.D1 (CLAUDE.md + hooks) — after C1 (makes ask available as "memory")
```

**Suggested sprint:**
1. B1 + B2 — config changes, immediate improvement (5 min)
2. A1 — `--max-items` flag (medium effort, highest impact)
3. A2 + A3 — slim frontmatter + body-first packing (medium effort)
4. C1 — `ask` command (medium effort)
5. D1 — CLAUDE.md + hooks integration (low effort, completes the loop)

## Acceptance Criteria

- [ ] `context-pack --max-items 10 --budget 4000` returns ≤10 items, most with body text
- [ ] `context-pack --slim` reduces frontmatter to {type, title, status}
- [ ] `memory recall` defaults to `--min-confidence medium` (tag_overlap excluded)
- [ ] `vaultmind ask "query"` returns search hits + packed context in one call
- [ ] CLAUDE.md instructs agent to use VaultMind for knowledge queries
- [ ] `vaultmind ask` works as a single-command knowledge lookup
- [ ] `task check` passes, coverage ≥ 85%
- [ ] All existing tests still pass (backward compatibility)
